# Project Execution Summary

This document summarizes the overall execution flow of the Automated Auditorium Lighting system, connecting all entry points to form the complete pipeline.

## Project Start Points
The project has a few orchestration entry points depending on the goal:
- `backend/pipeline_runner.py` → Used by the backend API to orchestrate the core automated lighting pipeline end-to-end (Phases 1 through 4) asynchronously.
- `main.py` → A CLI entry point meant primarily for triggering Phase 1 (script to scenes extraction).
- `phase_5/server.py` or `backend/app.py` → Start visualization/dashboard web servers.

## Phase Connection & Execution Flow

1. **Phase 1: Parsing & Segmentation**
   - **Input:** Raw script document (.txt, .pdf, .docx).
   - **Process:** Through `phase_1/__init__.py::run_phase_1`, the system acquires text, structures it immutably, chunks it, and feeds it to an LLM to segment logical scenes. Timestamps and validation checks are strictly applied.
   - **Output:** A schema-conformant JSON containing an array of structured "Scenes".

2. **Phase 2: Emotion Analysis**
   - **Input:** Scene texts generated by Phase 1.
   - **Process:** Passed into `phase_2/__init__.py::analyze_emotion`. Uses HuggingFace APIs (Llama 3 Instruct) or a DistilRoBERTa fallback classifier to assign primary/secondary emotion profiles.
   - **Output:** The scene dict enriched with a strictly typed `emotion` payload.

3. **Phase 3: Knowledge Layer (Dual RAG)**
   - **Input:** Evaluated emotions (from Phase 2) and script parameters.
   - **Process:** Handled by `phase_3/__init__.py::get_retriever()`. Looks up matching lighting design rules (semantics) and hardware capabilities (auditorium fixtures) from local FAISS vector databases.
   - **Output:** Returns prompt context blocks mapping emotions to lighting semantics.

4. **Phase 4: Lighting Decision Engine**
   - **Input:** Scene text + Timing + Emotion (Phase 1 & 2) combined with the Retrieval Context (Phase 3).
   - **Process:** Executed by `phase_4/lighting_decision_engine.py::LightingDecisionEngine`. It evaluates the design rules and outputs abstract "intent" (e.g., color `warm_amber`, intensity `80%`, transition `fade`) instead of hard DMX code.
   - **Output:** A `LightingInstruction` object detailing semantic intent for logical fixture groups (front_wash, specials, etc.).

5. *Interjection* - **Phase 6: Orchestration Validation**
   - **Input:** Generated cues from Phase 4.
   - **Process:** `phase_6/cue_validator.py` applies logical checks to ensure instructions match the physical parameters defined in Phase 3 databases.
   - **Output:** A validity check preventing malformed lighting code execution.

6. **Phase 5: Simulation & Visualization**
   - **Input:** Validated `LightingInstruction` arrays over time.
   - **Process:** Provided sequentially to `phase_5/playback_engine.py` which pushes the states to a server (via WebSockets in `phase_5/server.py`). The `ThreeJSAdapter` converts abstract parameters to 3D rendering data.
   - **Output:** JSON-encoded 3D visualization packets broadcasted to frontend web clients.

7. **Phase 7: Evaluation**
   - **Input:** Pipeline input/output pairings.
   - **Process:** Passively logs decisions (via `phase_7/trace_logger.py`) and calculates offline mathematical metrics related to consistency and deterministic stability (via `phase_7/metrics.py`).
   - **Output:** Offline JSON trace logs and metric report dictionaries.

8. **Phase 8: Hardware Translation (Future)**
   - **Input:** Visual states or semantic intents from Phase 4/5.
   - **Process:** [Unimplemented] Will convert semantic variables to 0-255 DMX byte data.
   - **Output:** Raw DMX control signals (e.g., Art-Net, sACN).

## Project Ending Points
The project theoretically terminates execution at the end of **Phase 5's Real-Time Playback** or when offline evaluation sequences successfully write execution traces in **Phase 7**. In production, it will ultimately terminate in **Phase 8**, physically actuating lighting hardware.
